TCP是什么

TCP是面向连接的,可靠的,基于字节流的传输层通信协议,具备传输可靠,数序一致,拥塞控制等特性

TCP和UDP区别

对比项 TCP（Transmission Control Protocol） UDP（User Datagram Protocol）
连接方式 面向连接（需要三次握手） 无连接，随发随走
传输可靠性 可靠：保证顺序、无重传、无丢失 不可靠：可能丢包、乱序、不重传
数据边界 无边界，面向字节流（需应用层划分消息） 有边界，面向报文（1次发送=1次接收）
传输效率 较低（连接建立、确认应答、重传机制） 高效（无需握手、确认、重发）
开销大小 较大（需要维护连接状态、滑动窗口、ACK） 小（只加了一个 8 字节头部）
是否有拥塞控制 有（慢启动、拥塞避免等算法） 无，自行控制
应用场景 文件传输、HTTP、数据库、邮件（要求可靠） DNS、视频直播、语音通话、游戏（要求实时）
常用端口示例 80（HTTP）、443（HTTPS）、22（SSH） 53（DNS）、67/68（DHCP）、69（TFTP）
消息边界是指一段数据的起止范围是否天然被保留,协议是否知道这是一条完整信息

为什么要TCP,IP层实现控制不行吗

IP是网络层协议,尽力而为
不保证可靠,不保证顺序一致,不保证包不会重复或者丢失
仅仅提供地址寻址 + 路由转发

IP层设计成无状态,尽量简单

  1. 这符合分层设计原则
  2. 并且IP是面向节点的,TCP是端到端的,中间节点不需要知道这些信息,也不应该参与
  3. 不是所有上层应用都需要可靠传输,如果把这些功能都放在IP层,上层应用性能会出现问题

TCP粘包和拆包

粘包: 多个发送方的数据,在接收方被一次性读到了一起,合并成了一个包,边界丢失
拆包: 一次发送的数据,在接收方被拆成了多次读取,接收方只收到一部分

发生场景

粘: 客户端发送多个小数据包,服务器一次read全读进来
拆: 服务器发送的数据过大,或者网络拆分,客户端一次read只能收到部分
粘+ 拆: 高速并发时,两个包半粘 半拆

原因

TCP是流式协议,没有消息边界的概念
加上系统缓冲区的传输效率优化,一收一发在TCP里是不存在保证的

解决方案

固定长度协议,每条消息N字节
优点:处理简单
缺点:灵活性差

添加长度字段
[总长度][消息体]
最常用,客户端先发长度,再发消息实体

分隔协议符

类似HTTP里\r\n,但是不能用于二进制或内容中有分隔符的情况

TCP三次握手

客户端首先发起一个SYN给服务端, SYN_SENT

服务端收到SYN后,发送SYN-ACK给客户端,SYN_RCVD

客户端收到SYN-ACK后返回一个ACK, ESTABLISHED

服务端收到ACK建立成功, ESTABLISHED

ISN初始序列号是怎么取值的

ISN是连接建立时选定的第一个序列号
比如建立三次握手时,发送的SYN_SENT携带seq=x

ISN用来标识每个字节的编号,用于确认接受范围,避免混乱连接

初始设计(RFC 793)
ISN是一个32位无符号整数,每4微妙自增1, 每秒自增250000

每次新建连接取当时计数器作为ISN,每个主机维护一个全局ISN

安全加强版本(RFC 1948)

为了防止TCP序列号预测攻击,ISN的生成加入了随机性 + 哈希算法
ISN = MD5(src_ip, dst_ip, src_port, dst_port, secret_key, time_tick)
加入4元祖 密钥 时间戳
保证唯一

TCP三次握手时,发送SYN后宕机了会怎么样

服务端如果没有收到SYN,就没有下一步了
服务端如果收到了SYN,就会发送SYN+ACK, 实际上创建了一个SYN_RCVD半连接状态,把连接信息放入了半连接队列

超时时间计算RTO = SRTT + max(G, 4 * RTTVAR)

客户端已经宕机,无法做出对SYN的ACK,就会重传SYN+ACK包,在重传到默认5次后,服务端把这个连接从队列中移除,状态变成closed 资源释放

SYN Flood攻击

由此引出SYN Flood,这是一种拒接服务攻击

就是利用上述的 客户端发送SYN(伪造IP),然后不返回ACK

这样占满半连接队列,正常用户就无法响应导致连接失败
服务器资源耗尽

防御方法

  • SYN Cookie 不保存半连接状态,把信息编码进ISN,收到ACK再建连接
  • 限制半连接数
  • 防火墙,限制单位时间内的SYN请求数量
  • 使用代理,少暴露后端真实IP
  • 启用TCP syncache 连接缓存机制

TCP四次挥手

第一次挥手: 客户端发送FIN 主动关闭连接, 进入FIN_WAIT_1
第二次挥手: 服务端收到FIN之后,不再接受数据,发送ACK包,进入CLOSE_WAIT,客户端收到FIN进入FIN_WAIT_2
第三次挥手: 服务端处理完最后的数据后,服务端发送FIN,进入LAST_ACK
第四次挥手: 客户端回复ACK, 进入TIME_WAIT,等待2MSL,之后CLOSED, 服务端收到客户端ACK,也进入CLOSED

TCP是全双工通信,连接的关闭必须双方各自发送FIN包

当没有数据处理的时候 FIN+ACK可以一同由服务端返回,也就是三次挥手,这是符合设计规范的

TIME_WAIT存在的必要

  • 保证最后一个ACK能被对方接受
    • 如果ACK丢失,服务端会重新发送FIN
    • 客户端还在,能重发ACK确保连接正常关闭
  • 防止旧连接的延迟数据污染新连接
    • 如果立刻释放端口,旧数据的包可能误导新连接

TCP是通过四元组识别的,如果关闭连接后,网络中还有一些没送到的迟到包,会被识别成合法数据表,导致数据错乱,这个能保证网络中所有旧报文都已经消失

服务端不会回复ACK,即使客户端返回的ACK丢失,服务端重传FIN也丢失的话,超过2msl,客户端进入closed,服务端重发FIN超时后自己关闭连接

除了四次挥手,还有什么方法能关闭链接

  1. RST标志: 不保证已经发送到数据传输完成,对端收到后立刻断开连接 Connection reset by peer
    1. 向已经关闭的连接发送数据
    2. 服务端调用close前未处理读写(例如直接close(fd))
    3. 程序异常崩溃或者被kill
    4. 接到非预期序列号数据
  2. TCP超时
  3. 调用shutdown半关闭连接
  4. 操作系统层断开(拔网线等)

TCP超时重传机制是为了解决什么问题; 重传策略

拥塞控制的步骤

常规TCP拥塞控制

  1. 慢启动
  2. 拥塞避免
  3. 快速重传+快速恢复
  4. 超时重传

核心是cwnd,ssthresh

慢启动

初始cwnd = 1
每收到一个ACK, cwnd+=1

1 -> 2 -> 4 -> 8…

拥塞避免

当cwnd >= ssthresh

改成线性增长
cwnd += 1/cwnd

就是每次总加一

快速重传 + 快速恢复

收到3个重复ACK 说明某个包丢了
立刻重传丢失的包,ssthresh = cwnd / 2
cwnd = ssthresh + 3
重新进入慢启动

超时重传

没有收到ACK,认为网络严重拥塞

ssthresh = cwnd / 2
cwnd = 1

回退重新开始

Cubic

默认用于linux

cwnd = C(t - K)^3 + origin_point

发生拥塞之后不是减半,而是尝试平滑的回升

t是举例上次拥塞事件的时间
k是使cwnd恢复到上次最大值的时间点
origin_point是上次丢包时的cwnd

BRP

谷歌提出的
核心思想是 不以丢包为信号,而是以瓶颈带宽 + 最小RTT作为核心指标

四个阶段

  • startup 快速探测最大带宽(指数增长)
  • drain 排空队列,降低发送速率
  • probeBW 维持带宽检测 尝试微调
  • preobeRTT 周期性减速以重新测量RTT

BDP = 带宽 × RTT

TCP 有超时重传为什么还需要快速重传机制

操作系统

操作系统在进行线程切换时需要进行哪些动作

发生线程切换的场景

  1. 时间片耗尽
  2. 主动让出CPU (等待锁,资源, IO等)
  3. 高优先级线程就绪
  4. 系统调用阻塞(read)

线程切换的本质 :保存当前上下文 + 加载另一个线程的上下文

上下文就是寄存器值 + 栈指针 + 程序计数器 + CPU状态寄存器 + 一些调度器元信息

关键操作

  1. 触发切换
  2. 保存当前线程的上下文信息
    1. 保存寄存器(PC程序计数器, SP栈指针, 通用寄存器)
    2. 保存浮点寄存器
    3. 保存当前线程状态到TCB
    4. 设置TASK_INTERRUPTIBLE / TASK_RUNNING等状态
  3. 内核调度器选择下一个要运行的线程
    1. 调度器根据调度算法来选线程
  4. 加载目标线程的上下文
    1. 从目标线程的TCB中恢复寄存器值
    2. 设置任务状态为TASK_RUNNING
    3. 切换内核栈
  5. 更新MMU/页表
    1. 如果目标线程属于另一个进程,还需要切换页表
  6. 恢复CPU状态, 跳转回用户空间继续执行

进程的调度算法

算法名称 英文简称 是否抢占式 主要特点说明
1. 先来先服务 FCFS ❌ 非抢占 简单易实现，可能导致“长作业阻塞短作业”
2. 短作业优先 SJF ❌ 非抢占 最低平均等待时间，但需提前知道执行时间
3. 最短剩余时间优先 SRTF ✅ 抢占式 SJF 的抢占版本，动态适应短任务
4. 时间片轮转 RR ✅ 抢占式 每个进程轮流运行固定时间片，公平但上下文切换多
5. 优先级调度 Priority ✅ or ❌ 按优先级高低调度，可饿死低优先级进程
6. 多级反馈队列 MFQ ✅ 抢占式 结合 RR+Priority+动态优先级，适合多类型任务调度
7. 实时调度 EDF、RM ✅ 抢占式 用于实时系统任务调度，如 deadline 优先或固定周期任务

FCFS

先来先服务

进程按照到达时间进入就绪队列,当前进程未完成就不会被切换

SJF

短作业优先,每次从队列里挑选剩余执行时间最短的进程

SRTF

最短剩余时间优先

抢占式的SJF
每次新进程到达,比较当前正在运行的和新来的谁剩余更短
如果新来的剩余更短,就抢占当前进程

RR (Round Robin)

时间片轮转
每个进程按顺序轮流执行一个时间片,超过时间片就放入队尾,就绪列队通常是FIFO

优先级调度

选择优先级最高的进程
支持抢占式和非抢占式

多级反馈调度

多个队列 Q0 -> Q1 -> Q2
初始进程进入Q0
如果当前时间片没有执行完,降级到下一级队列
新进场总是先插入到高优先队列,通常为抢占式

实时调度

EDF

每次从就绪列队里选择最早截止时间到任务来运行

抢占式调度,动态优先级
调度最优:只要总负载小于100%,就能完成所有任务

RM

每个周期性任务的优先级, 周期越短,优先级越高

静态优先级调度
抢占式,容易实现,但是调度利用率不如EDF

CFS

记录虚拟运行时间vruntime,vruntime越小代表运行的少,更优先调度,每运行一段时间,就累加vruntime, 优先级高的进程加的速度更慢

使用红黑树来维护所有就绪进程,左边是vruntime最小的进程,每次调度都取树的最左边节点执行

什么是软中断、什么是硬中断

中断是指 处理器暂停当前任务,转而去响应某个事件的机制

硬中断

由硬件设备产生的中断信号,通知CPU某个外设事件已经完成,需要处理

比如

  • 键盘输入
  • 硬盘读写完成
  • 网卡收到数据包
  • 定时器中断
    这种中断是异步的,CPU不知道什么时候发起中断,但是中断发生时,会立刻终止当前执行,保存上下文,转而去执行中断服务程序

软中断

由软件指令触发的中断,一般用于操作系统调用,调式陷阱等

  • 用户执行系统调用 syscall
  • kill(pid, SIGUSR1) 发送软件中断信号
  • 内核中softirq, tasklet是软中断的进一步应用

是同步的,程序代码主动触发,切换到中断处理程序

linux的软中断有一种特殊的含义,是在硬中断上下文中延迟执行的任务机制
包括

  • softirq: 轻量级软中断, 网络协议栈, 块设备等
  • tasklet: 基于softirq的抽象
  • workqueue: 可调度延后执行,运行在进程的上下文

为什么要有虚拟内存

  1. 进程隔离
    1. 没有虚拟内存时,多个程序直接访问物理地址,容易互相篡改,破坏数据
    2. 虚拟内存为每个进程提供独立的地址空间,使得进程之间互不干扰
  2. 地址空间拓展
    1. 程序可以认为自己有很大的内存,即使物理内存很小
    2. 看起来很大的内存,运行大型程序成为可能
  3. 内存复用与共享
    1. 多个进程可以映射到同一块物理内存,节省资源
  4. 支持内存换页,把暂时不用的内存内容换出到磁盘,腾出物理内存
  5. 简化内存分配与管理
    1. 虚拟内存支持非连续性物理页映射给连续虚拟地址,方便内存分配器管理

虚拟内存解决了以下问题

  • 进程互相干扰
  • 内存不足
  • 内存碎片
  • 程序移植困难
  • 难以共享内存

什么是分段, 什么是分页

分页是把内存 按固定大小 切分，便于管理，解决 内存碎片 问题
分段是把程序 按逻辑结构 切分（如代码段、数据段），更符合程序员视角，利于 权限隔离

分页

将虚拟内存和物理内存都划分成固定大小的块,虚拟块叫页,物理块叫页框

页大小固定,通常是4KB
分配和回收内存以页为单位,便于管理不易产生外部碎片
页之间无逻辑结构意义

虚拟地址 = 页号 + 页内偏移
CPU ➝ 通过页表将页号映射到物理页框号，再加偏移组成物理地址

分段

将程序按照 逻辑结构 分为若干段，每段有自己的 基址（Base） 和 长度（Limit）

例如：
代码段（.text）
数据段（.data）
堆段（heap）
栈段（stack）

段大小不固定,每段都有一个起始地址 + 长度,便于访问控制和内存保护,可能产生外部碎片

逻辑地址 = 段号 + 段内偏移
通过段表：段号 ➝ 基址 + 偏移，形成物理地址

分页机制是主流的内存管理方式

CPU 使用率和CPU负载指的是什么

CPU 使用率是指单位时间内, CPU实际处于忙碌状态的百分比 比如 70% 表示 CPU 有 70% 的时间在处理任务

CPU负载是指单位时间内，系统中处于运行队列或等待 CPU 的进程数（也包括部分等待 I/O 的进程）

CPU使用率高 CPU负载高 说明CPU忙且很多进程在排队，需要优化程序/加核数
CPU使用率高 CPU负载低 说明CPU在忙，但没有排队，系统压力小
CPU使用率低 CPU负载高 说明很多进程在等待（如I/O），但CPU空闲，可能是磁盘或网络瓶颈

I/O是什么

I/O 是 Input/Output 的缩写，在计算机中指 数据在不同组件之间的传输过程。在网络编程中，指的是：

• Input：从网络中读取数据

• Output：向网络中写入数据

用户调用 read() 读取 socket 数据时，内核可能还没准备好数据

如果采用的是 阻塞式 I/O（Blocking I/O），这个系统调用会一直 阻塞等待数据准备好，期间线程无法做其他事
或者系统资源有限,在等待
