Redis

发表于2025-03-03|更新于2025-03-03
|字数总计:2.1k|阅读时长:6分钟

简介

Redis是一个基于内存的数据库,对数据的读写操作都是在内存中完成的,因此读写速度非常快,常用于缓存 队列 分布式锁等场景

Redis有多重数据类型,比如String, Hash, List, Set, Zset, Bitmaps, HyperLogLog等等

和Memcached的区别

共同点 :

  1. 都是基于内存的数据库, 一般用来当做缓存使用
  2. 都有过期策略
  3. 性能非常高
    区别:
  4. Redis的数据类型更丰富,后者只支持简单的key-value
  5. Redis支持数据的持久化,可以将内存中的数据保持在磁盘中,重启的时候可以再次加载使用,后者重启或挂掉就没了
  6. Redis原生支持集群模式,后者只能依靠客户端实现往集群分片中写入数据
  7. Redis支持发布订阅模型,lua脚本,事务等功能

为什么用Redis作为Mysql的缓存

主要是因为他具备高性能和高并发两种特性

前者很好理解,第一次读取数据库后,把数据缓存在Redis,之后直接命中内存的Redis,速度非常快

后者单台设备Redis的QPS能轻松破10w

数据结构



后续又支持四种数据类型

BitMap: 二值状态统计的场景,如签到,用户在线状态等
HyperLogLog: 海量数据基数统计的场景
GEO: 存储地理位置信息的场景
Stream: 消息队列,自动生成全局唯一消息ID,支持以消费组形式消费数据

String

String类型的底层数据结构是SDS,简单动态字符

  1. SDS不仅可以保存文本,还能保存二进制数据(如图片,音频,等)
  2. SDS获取字符串长度的时间复杂度是 O(1)
  3. Redis的SDS API是安全的,不会造成缓冲区溢出

List

List 底层数据结构是双向链表或压缩列表
如果列表的元素小于512个,列表的每个元素的值都小于64字节,就使用压缩列表做底层
否则使用双向链表

但是在Redis3.2之后,无论什么情况都只用quicklsit

Hash

底层数据结构是压缩列表或哈希表

同样在Redis7中,压缩列表已经被废弃,交给listpack来实现

Set

底层是哈希表或整数集合实现的

如果集合中的元素都是整数且元素个数小于512, Redis使用整数集合作为Set类型的底层数据结构
否则使用Hash表

ZSet

底层数据结构是压缩列表或者跳表

同样在Redis7中,压缩列表已被废弃

Redis线程模型

Redis是单线程吗

Redis的单线程指的是 接收客户端请求 -> 解析请求 -> 进行数据读写 -> 发送数据给客户端
这个过程是一个线程完成的,但是Redis本身还有启动后台线程

  1. Redis在2.6之前,会启动两个后台线程,分别处理关闭文件,AOF刷盘
  2. 4.0后,新增一个lazyfree线程,用于异步释放Redis内存
    这些任务是十分耗时间的,如果放在主线程处理,很容易发生阻塞

关闭文件,AOF刷盘和释放内存都有各自的任务队列

单线程模式

6.0之前的单线模式

蓝色部分用主线程负责的事件循环

首先调用epoll_create()创建一个epoll对象和调用socket()创建一个服务端socket
然后,调用bind绑定端口和listen监听这个Socket
再将调用epoll_ctl()把listen Socket加入到epoll,注册连接事件处理函数

初始化之后就是事件循环

  • 调用处理发送队列函数,如果发送队列有发送任务,就调用Write将客户端发送缓存区里的数据发送出去,如果一次没发送完,注册写事件处理函数等待epoll_wait发现事件
  • 调用epoll_wait
    • 如果是连接事件,调用accept获取已经连接的Socket,调用 epoll_ctl 将已连接的 socket 加入到 epoll,之后注册读事件处理函数
    • 如果是读事件,调用读事件处理函数 调用read获取客户端发送的数据 -> 解析命令 ->处理命令 -> 将客户端对象添加到发送队列 -> 将执行结果写到发送缓存区等待发送
    • 如果是写事件,调用写事件处理函数 通过write把客户端发送缓冲区里的数据发送出去,如果一次没发送完,注册写事件处理函数等待

为什么Redis使用单线程还这么快

  • 大部分操作在内存中完成,而且使用高效的数据结构, Redis的瓶颈可能是机器的内存或者网络带宽
  • 避免多线程之间的竞争,省去了多线程切换的时间和性能损耗,也没有死锁问题
  • 采用 I/O多路复用机制, 也就是 select / epoll机制, 它允许在Redis运行单线程的情况下,允许在内核中有多个监听Socket和已连接Socket 一旦有请求到达,就会交给Redis线程处理,这就实现了一个Redis线程处理多个IO流的效果

为什么6.0之后要引入多线程

这是因为随着网络硬件的性能提升,Redis的性能瓶颈也可能出现在网络I/O处理上

Redis6.0对于网络I/O采用多线程来处理,但是对于命令执行,仍然还是单线程

Redis6.0版本支持的I/O多线程特性,默认I/O多线程只针对发送响应数据,不会已多线程的方式处理读请求
要想打开可以在配置文件里修改

在Redis6.0后,Redis在启动的时候,默认会额外创建6个线程(不包括主线程)

  1. Redis-server 主线程
  2. bio_close_file, bio_aof_fsync, bio_lazy_free
  3. io_thd1, io_thd2, io_thd3 默认线程是4,所以会启动3个多线程

持久化

如何实现数据不丢失呢,Redis有三个办法

  • AOF日志
  • RDB快照
  • 混合持久化方式

AOF日志

Redis每执行完一条写操作,就把该操作追加到一个日志文件里去
能根据AOF文件恢复

先执行写操作 再写入AOF日志
可以避免额外的检查开销,避免第一次命令错误 同时这不会阻塞当前写操作的命令执行

风险是 数据可能丢失, 没来得及写入硬盘就挂了, 还可能阻塞其他操作

写回策略

三种写回策略

  • Always
  • Everysec
  • No

AOF重写

当AOF日志过大的时候,恢复数据的过程就会异常慢

所以为了避免AOF文件越写越大,提供了AOF重写机制,当AOF超过这个大小的时候,就读取当前数据库所有键值对,然后把每一个键值对用一条命令记录到新AOF文件里去

重写的过程中是由bgrewriteaof完成的

子进程在AOF重写期间可以继续处理命令,子进程带有主进程的数据副本,这里不使用线程的原因是多线程共享内存,在修改共享内存的时候需要加锁,会降低性能

触发重写机制后,父子进程共享物理内存,重写子进程只会对这个内存进行只读,重写AOF文件

在重写过程中主进程依旧可以正常处理命令,如果主进程修改了已经存在的kv,那么会发生写时复制,Redis设置一个AOF缓冲区,这个缓冲区在创建bgrewriteaof后子开始使用

在重写AOF期间,当Redis执行完一个写命令后,会将写命令写入到AOF缓存区和AOF重写缓冲区

子进程完成AOF重写工作,向主进程发送一条信号
主进程收到信号后,调用一个信号处理函数,主要完成

  1. 将AOF重写缓冲区的内容追加到新的AOF文件中,使得新旧两个AOF文件保存的数据库状态一致
  2. 新的AOF文件进行改名,覆盖现有的AOF文件
